# Notification 아키텍처 설계

## 핵심 원칙
사용자에게 아무것도 시키지 않는다.
설정에서 토글 하나로 켜고 끌 수 있으면 충분.

---

## 채널 결정

| 채널 | 사용자 액션 | 결정 |
|------|-------------|------|
| personalEmail 알림 | 없음 (이미 데이터 있음) | ✅ 메인 |
| Telegram | /start 최소 1회 필요 | 🔲 선택적 opt-in |
| 카카오/슬랙/앱 | 설치·인증 필요 | ❌ 보류 |

---

## 구현 설계

### 트리거 포인트
- 외부 수신 → `mailer-worker/src/index.ts` Firestore 저장 직후
- 내부 수신 → `src/app/api/send/route.ts` 내부메일 저장 직후

두 곳 모두 `notify(to, { from, subject })` 하나만 호출.

### Firestore 구조
```
members.{email}:
  notifications:
    emailEnabled: boolean   // 기본 false, 설정에서 토글
    telegramChatId: string  // 선택, 직접 입력
    telegramEnabled: boolean
```

### notify.ts 인터페이스
```typescript
// src/lib/notify.ts
export async function notify(
  recipientEmail: string,   // e.g. "sky@ourim.kr"
  mail: { from: string; subject: string }
): Promise<void>
```

내부에서 members 문서 조회 → enabledChannels만 발송.

### personalEmail 발송
Resend 사용 (현재 5명 기준 쿼터 여유 충분).
사용자 증가 시 WP 서버 sendmail로 교체 가능.

발송 내용:
```
제목: [ourim.kr] 새 메일이 도착했습니다
본문: {from}님으로부터 메일이 도착했습니다.
      제목: {subject}
      → gw.ourim.kr 에서 확인
```

### Telegram (선택 opt-in)
1. 관리자가 BotFather에서 봇 생성 → `TELEGRAM_BOT_TOKEN` 설정
2. 사용자가 원하면 설정 페이지에서 chat_id 직접 입력 (봇에서 /start 후 확인)
3. 아무도 안 써도 시스템에 영향 없음
